# Agent README

이 문서는 Cursor 세션이나 대화 컨텍스트가 날아갔을 때, 다음 에이전트가 현재 프로젝트 상태와 주의점을 빠르게 파악하기 위한 작업 메모입니다.

## 프로젝트 개요

- 작업 루트: `f:\AI 채팅`
- 주요 프로젝트: `f:\AI 채팅\lorebox-worker`
- 배포 URL: `https://lorebox-worker.levantarlos.workers.dev`
- 스택: Cloudflare Worker + URL 파라미터 기반 SVG 렌더러
- 목적: Lunatalk/캐릭터 채팅용 세계관 UI 이미지 생성
- 장르: 2036년 한국 현대 판타지, 경계관리청, 무속 기반 제령/봉인/천도/괴이 대응

## 구현된 테마

- `theme=alert`: 경고/공지 카드. 현재 경고창 디자인 톤은 좋은 기준점이다.
- `theme=dex`: 강령체/괴이 도감 DB. `dex.json` 등록 ID 기반으로 출력한다.
- `theme=shop`: 경계관리청 물자몰/상점 UI. 현재는 보류. 나중에 아이템 데이터가 정리되면 다시 다듬는다.
- `theme=arcade`: 인형뽑기/보상 카드.
- `theme=combat`: 전투 시작/전투중 카드. 현재 가장 민감한 작업 구간이다.

## 현재 Combat 기준

현재 배포본은 첫 빌드 구조를 기준으로 되돌린 뒤, 최소 수정만 반영한 상태다.

- `engage`: 첫 조우/전투 시작 카드. 높이 `174`.
- `resolve`: 전투중 판정 카드. 높이 `214`.
- 구조를 더 키우거나 16:9로 늘리지 말 것.
- `resolve` 화면은 현재 버전 그대로 유지하는 것이 좋다.
- 세그먼트 게이지는 3px 간격의 촘촘한 형태가 현재 기준이다.
- 색은 `alert` 경고창에 가까운 어두운 적색/금색 계열을 선호한다.
- 판정문에는 아주 약한 글로우만 허용한다.

## Combat 작업 주의

- 마음에 든 버전이 나오면 먼저 기준점으로 고정한다.
- 사용자가 특정 요소만 지적하면, 그 외 레이아웃/구조/비율을 바꾸지 않는다.
- "세로폭을 늘려라"는 명시가 없으면 높이를 늘리지 않는다.
- 16:9 확장은 다시 시도하지 않는다.
- `engage`와 `resolve`를 같은 구조로 억지 통일하지 않는다.
- `engage` 첫턴 카드에는 판정 결과, 진행도, 유저 상태를 기본 표시하지 않는다.
- `resolve`는 한자 마크 + 판정 라벨 + 판정문 + 상대 임무 게이지 + 유저 영력 게이지 구조를 유지한다.
- 텍스트를 중앙정렬로 바꾸지 않는다. 특히 `resolve`의 라벨/판정문은 좌측 기준 흐름을 유지한다.
- 박스 폭, 여백, 위치를 "비슷하게" 재현하려고 새 좌표를 만들지 말고, 기존 안정 버전을 기준으로 수정한다.

## 용어/세계관 주의

- 한국 무속 기반 괴담/제령물 톤을 유지한다.
- 사용 가능한 핵심어: `영력`, `신기`, `신력`, `부적`, `무구`, `살풀이`, `정화`, `제령`, `퇴치`, `봉인`, `천도`, `원한`, `사념`, `한기`, `영혼 경계`, `빙의된 기척`.
- `HP/MP`, `마나`, `회로`, `오염+N`, `체력 감소`, `봉인식`, `분리`, 일본식 요괴/주술 용어는 피한다.
- `부적 전개`보다 `부적 사용`이 낫다.
- `자아침식`은 상위 귀신, 빙의, 강한 사념 접촉 같은 트리거가 있을 때만 표시한다.
- Worker 안에 AI 지침이나 금지어 정규화를 넣어도 나레이터 AI는 읽지 못한다. Worker는 렌더러일 뿐이다.

## 최신 세계관 기준

- 현재 기준 연도는 `2036년`이다.
- 경계 이상과 강령체 사건은 최근 갑자기 생긴 일이 아니라 `2020년대부터 누적된 재난`이다.
- 2036년 현재 경계관리청과 창사 제도는 이미 제도화되어 있다. 베테랑 NPC가 "오래 굴렀다"고 말해도 설정 충돌이 나지 않아야 한다.
- 한국의 핵심 무대는 `부산 해운대 경계관리청 부산지부`다.
- 작중에서 무속적 힘을 쓰는 사람은 전부 `창사`라고 부른다. `헌터`, `능력자`, `퇴마사`는 작중 호칭으로 쓰지 않는다.
- 단, 소개글/홈페이지에서는 유저 이해용 비유로 "헌터물의 게이트/레이드 구조를 한국 무속 기반 창사·제령 체계로 재해석" 정도는 사용 가능하다.
- `주문`, `결박`, `봉인`은 사용 가능하다. 문제는 `봉인식`, `술식`, `결계식`처럼 일본식 판타지 어감으로 붙는 표현이다.

## 문/개문 설정

- `경계 이상`: 현실의 지형, 공기, 소리, 습도, 공간감이 어긋나는 전조. 작은 사건과 조사 장면에 사용한다.
- `문` 또는 `경계의 문`: 경계가 무너져 생긴 위험 지점. 헌터물의 게이트에 가깝지만 작중에서는 게이트라고 부르지 않는다. 형태는 한국 사찰의 산문처럼 보이는 방향이 좋다.
- `개문`: 문이 실제로 열려 원혼·강령체가 쏟아지는 상태. 게이트 오픈/몬스터 웨이브 포지션.
- `명계의 문`: 일반 돌발 사건에 쓰지 않는다. 시왕/염라 관련 메인 챕터급 대형 재앙으로 제한한다.
- `용궁`: 부산·해양·수몰·해저 경계 관련 대형 사건에 제한적으로 사용한다.
- 일반 돌발마다 문을 열지 않는다. 평상시 사건은 `경계 이상`, `미확인 문`, `개문 전조`, `민간인 빙의 의심`, `주민 신고`, `도감 미등록 반응` 위주로 처리한다.

## 루나톡 프롬프트 구성 기준

- 플랫폼 메인 프롬프트는 5천자 내외가 한계다. 이미지팩 사용 시 4천자지만, 현재 SVG는 Worker 외부 링크라 이미지팩 용량과 분리된다.
- 메인 프롬프트에는 매턴 필요한 것만 둔다: 세계관 압축, 용어, 시간 연속성, 상태창, 초반 고정 사건, SVG 출력 원칙.
- 상세 연표, 홍보문, 시스템 상세 설명은 홈페이지/소개페이지로 뺀다.
- 로어북은 동시에 최대 4개만 활성화될 수 있으므로 우선순위를 강하게 둔다.
- 로어북/플랫폼 프롬프트는 "플랫폼 AI가 실제로 읽고 수행할 수 있는 것"만 기준으로 쓴다.
- Worker가 읽는 파일(`dex.json`, PNG 경로 등)을 플랫폼 AI가 직접 조회할 수 있는 것처럼 쓰지 않는다.
- 잘못된 예: "dex.json ID만 사용", "DB를 조회", "파일을 읽어라".
- 올바른 예: "도감용 로어북의 BMA 목록만 참조", "목록에 없으면 BMA-0180(미확인)".
- 추천 로어북 4개:
  1. `[도감·강령체 DB]`
  2. `[전투·제령 판정]`
  3. `[돌발 사건·문·개문]`
  4. `[상점·뽑기·보상 UI]`
- NPC 상세는 당장 4개 안에 넣기 어렵다. 청연/채서하 핵심만 메인에 1~2줄로 두고, 상세는 소개페이지나 추후 로어북으로 분리한다.

## 루나톡 소개문 HTML/CSS 확인 기준

- 사용자가 개발자도구로 확인한 루나톡 캐릭터 상세 페이지 구조:
  - 소개 영역은 대략 `section.detail > div.cont > div.inner > div.detailInfo` 아래에 위치한다.
  - 상세 소개 본문은 `div#charDescriptionHost` 안에 렌더링된다.
  - `charDescriptionHost` 내부는 `#shadow-root (closed)`로 격리되어 보였다.
- 캡처 기준으로 Shadow DOM 내부에 `<style>` 태그와 `style="..."`가 살아 있었다.
- 따라서 루나톡 소개문에는 최소한 일반 HTML, `<style>`, 인라인 CSS, `div`, `h3`, `br`, `hr`류 태그가 작동할 가능성이 높다.
- 배경색, 테두리, 그라데이션, flex 레이아웃 같은 CSS도 실제 소개문 영역 안에서 적용된 흔적이 있었다.
- 단, JS, 외부 스크립트, 복잡한 슬라이더, 클릭 인터랙션은 아직 확인되지 않았다. 루나톡 소개문에는 넣지 않는 쪽이 안전하다.
- 루나톡 소개문은 `상단 대표 SVG 1장 + 핵심 소개문 + 2~3개 미리보기 카드 + 접이식 상세` 정도로 안정적으로 구성한다.
- 슬라이더, 강한 애니메이션, 더 화려한 후킹 페이지는 루나톡 소개문 안에서 억지로 구현하지 말고 GitHub Pages 소개 사이트에서 처리한다.
- 역할 분리:
  - 루나톡 소개문: 빠르게 읽히는 홍보/입문 페이지. 안정성 우선, 인라인 CSS와 단순 HTML 중심.
  - GitHub Pages: 시각적 충격과 자세한 미리보기용 랜딩 페이지. 슬라이더/섹션 전환/더 강한 연출 허용.
- 외부 SVG는 Worker URL을 `<img>` 또는 마크다운 이미지로 보여주는 방식이 가장 안전하다.
- 소개문 안의 버튼은 실제 조작 UI처럼 보이게 만들더라도 클릭 기능을 전제하지 않는다. 필요한 경우 GitHub Pages 링크 버튼 정도만 둔다.

## 시작 도입 기준

- 기존 시작은 신입 창사 등록 후 관사/오리엔테이션 대기로 흘러 루즈해졌다.
- 출시용 시작은 등록 절차를 유지하되, 등록 직후 3~5턴 안에 반드시 첫 사건이 발생해야 한다.
- 권장 흐름:
  1. `{{user}}`가 부산 경계관리청에서 창사 등록 마지막 절차를 밟는다.
  2. 등록 직후 관리청 단말에 긴급 경고가 뜬다.
  3. 해운대 인근에서 경계 이상 또는 미확인 강령체 반응이 발생한다.
  4. 도감 조회는 `BMA-0180` 미확인 또는 이미 등록된 적절한 개체만 사용한다.
  5. 청연 또는 상관이 현장 보조 투입을 지시한다.
  6. 첫 대치는 전투 튜토리얼처럼 짧게 진행한다.
- 초반 목적은 설명보다 "관리청 시스템이 실제로 작동한다"는 인상을 주는 것이다.
- 초반 고정 사건에서는 `alert → dex → combat engage` 순서로 SVG를 한 번씩 자연스럽게 노출한다.

## 시간/돌발 규칙

- 사용자 메시지 1건 = 1턴. 보통 5~15분이 흐른다.
- 시간은 반드시 이전 상태창 이후로만 흐른다.
- 현재 시간보다 이른 시각으로 갱신해야 할 경우 반드시 다음 날짜로 넘긴다.
- 예: `6월 8일 목요일 15:00` → 전투 후 새벽이면 `6월 9일 금요일 02:00`.
- 금지: `6월 8일 목요일 15:00` → `6월 8일 목요일 02:00`.
- 돌발 사건은 상태창 날짜가 홀수이고 오후/야간 장면 전환이 있을 때만 1일 1회 후보로 검토한다.
- 중요한 대화, 감정 장면, 이미 진행 중인 전투/조사, 사용자의 질문에 답해야 하는 턴에는 돌발을 넣지 않는다.
- 일반 돌발에서 `명계의 문`을 반복 사용하지 않는다.

## 도감 최신 기준

- 도감 등급 체계는 `재해 / 1급 / 2급 / 3급 / 미확인`이다.
- `재해`는 1급을 대체하는 말이 아니라 1급 초과 별도 분류다.
- `이형`은 변이·뒤틀림 개체 전용 태그다.
- `1급 이형`은 가능하지만, `1급 재해 이형`처럼 등급을 겹쳐 쓰면 안 된다.
- 어색한 합성명은 피한다. `원혼집`은 폐기했고 현재 `BMA-0074`는 `참화원귀`다.
- Worker의 실제 도감 데이터는 `public/data/dex.json`이지만, 플랫폼 AI는 이 파일을 직접 읽지 못한다.
- 도감 로어북에는 AI가 참조할 수 있는 `[BMA 목록]`을 별도로 넣어야 한다.
- 도감 출력 지침은 "도감용 로어북의 BMA 목록만 참조한다. 목록에 없는 개체는 `BMA-0180` 또는 조사 중으로 처리한다"처럼 작성한다.
- 플랫폼 AI가 새 이름·등급·약점·사례를 임의 창작하지 않게 제한한다.
- 도감 URL은 BMA ID만 넘긴다. `name/desc/report`를 URL에 직접 쓰지 않는다.

## 인형뽑기 최신 기준

- `theme=arcade`의 S급은 캐릭터 인형 전용이다. 토끼/동물 인형은 A급 쪽이다.
- S급 캐릭터 인형 이미지는 `public/icons/npc/{iconId}.png`에 PNG가 있어야 뜬다.
- 예: `iconId=Doll_Seoha`를 쓰려면 `public/icons/npc/Doll_Seoha.png`가 실제로 존재하고 배포되어야 한다.
- 현재 로컬 `public/icons/npc`에는 README/labels만 있고 PNG가 없었다. 사용자가 다른 로컬 경로에 저장한 PNG는 승인 후 복사해야 한다.

## 디자인 선호

- 어두운 배경, 강한 대비, 얇은 선, 절제된 글로우.
- 과한 네온, 블러, 3D 엠보싱, 밝은 외곽선은 피한다.
- 박스를 그냥 나열하지 말고 실제 모바일 게임 UI처럼 정보 위계를 만든다.
- 명조 계열 한글 폰트를 선호한다. 기본 고딕/굴림 느낌은 피한다.
- 도감 UI와 경고창 UI의 톤앤매너가 좋은 기준점이다.
- 홀로그램/스캔라인은 은은하게만 사용한다. 배경이 텍스트보다 튀면 안 된다.

## 남은 논의거리

- 루나톡 메인 프롬프트를 `소개문 / 배경 세계관 / 시작 입장대사 / 시작 상황 관리 / AI 주의내용 및 로직` 양식으로 5천자 내 정리.
- 로어북 4개를 실제 트리거와 함께 작성: 도감, 전투, 돌발, 상점/뽑기.
- GitHub Pages용 소개 사이트 제작. Worker는 SVG API, GitHub Pages는 홍보/설정 홈페이지 역할.
- 썸네일은 캐릭터+타이포 중심, 소개글 상단은 `alert` SVG로 시스템 UI 차별점을 보여주는 구성이 좋다.
- `shop`은 아이템/아이콘 데이터가 더 준비된 뒤 다시 디자인 정리.

## 작업 원칙

- 파일 생성/삭제/이동은 사용자 승인 후 진행한다.
- 사용자가 명시적으로 "만들어줘"라고 지시한 파일은 승인으로 간주한다.
- 기존 기능 삭제, 레이아웃 변경, 디자인 변경은 지시된 범위 안에서만 한다.
- 사용자가 화가 난 상태에서는 변명하지 말고, 먼저 잘못을 인정하고 작업 범위를 좁힌다.
- 꼬이기 시작하면 새로 고치지 말고 안정 버전으로 롤백한 뒤 세부사항만 수정한다.
- 배포 후에는 반드시 실제 확인 링크와 버전 ID를 제공한다.
